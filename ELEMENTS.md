# ELEMENTS.md — Blueprint Elements Registry

Every element of the blueprint listed with a one-line description and a state.

**Legend:** ⬜ PROPOSED — not yet reviewed · ✅ VETTED — reviewed and confirmed by user

---

## Commands
Triggered by the user with `/name`. Claude executes the defined procedure.

> ⚠️ **Disambiguation risk**: if a plugin or MCP server also defines a command with the same name (e.g. `/onboard`), Claude may pick the wrong one. See R12 in REFACTOR.md for the open design decision on how to resolve this.

| | Element | Description |
|-|---------|-------------|
| ✅ | `/onboard` | Personalises the blueprint for the specific project: analyses the codebase, iterates until project purpose/goals/feature inventory are unambiguous, populates `.claude/knowledge/` (including `deferred.md` for backlog items); always run immediately after copying the blueprint |
| ✅ | `/status` | Dashboard snapshot of project health: fidelity coverage, requirements counts, open findings, escalation ages, sweep recommendation; read-only, no files changed, runs automatically at session start |
| ⬜ | `/checkpoint` | Creates a verified git commit: lightweight gate for spec-only changes (`.claude/` or spec files), full 7-check pre-commit gate for any source or test file change |
| ⬜ | `/integrity` | Active structural audit that searches for specific problems: broken links, stale docs, unregistered TODOs, spec inconsistencies; run on demand, not a dashboard — it finds what is broken |
| ✅ | `/docs` | Rebuilds user-facing `documentation/` folder and root `README.md` from `.claude/knowledge/` |

## Modes
Selected by Claude based on intent signals at the start of each response.

| | Element | Description |
|-|---------|-------------|
| ⬜ | Assess | Read-only status check; produces the project status report; no files changed |
| ⬜ | Feature | Full double-diamond flow for new features: requirements → design → implementation → verification |
| ⬜ | Design | Problem-space only: requirements gathering and design decision, no implementation |
| ⬜ | Bugfix | Test-first bug fix: reproduce → failing test → fix → verify → gates → checkpoint |
| ⬜ | Sweep | Fidelity audit: checks implementation against specs item by item, checkpointed per item |
| ⬜ | AdvSweep | Adversary-led sweep: finds attack surfaces and failure modes, prioritised by fidelity gaps |
| ⬜ | Review | Adversary role applied to a named artifact; targeted, not a full sweep |
| ⬜ | Audit | Auditor role applied to a specific named component; targeted, not a full sweep |
| ⬜ | Resume | Continues an interrupted Sweep from the last checkpoint |
| ⬜ | Ask | Used when intent is ambiguous; clarifies before selecting a mode |

## Roles
Behavioral contracts activated within modes. Each has a binding checklist.

| | Element | Description |
|-|---------|-------------|
| ⬜ | Analyst | Extracts requirements, maps domain concepts, surfaces unknowns and open questions |
| ⬜ | Architect | Produces design options with tradeoffs and consequences; converges on an ADR decision |
| ⬜ | Adversary | Attacks an artifact (design or implementation) across nine vectors; produces a findings list |
| ⬜ | Implementer | Writes code faithfully per the accepted ADR; no scope expansion |
| ⬜ | Auditor | Verifies a changeset against specs and invariants; produces a PASS/FAIL sign-off |
| ⬜ | Integrator | Verifies the change composes correctly with the rest of the system; produces deployment notes |

---

## Gates
Hard blocking checkpoints that must clear before forward progress.

| | Element | Description |
|-|---------|-------------|
| ⬜ | Adversary gate (design) | Runs after Architect; blocks any code until no CRITICAL findings remain |
| ⬜ | Adversary gate (implementation) | Runs after Implementer; blocks Auditor and commit until no CRITICAL findings remain |
| ⬜ | Auditor gate | Runs after Adversary clears; blocks commit until PASS sign-off is recorded |
| ⬜ | Pre-commit gate | Seven-check list (tests, sign-off, stale items, spec consistency, fidelity, references, local verification); blocks claiming done |

## Knowledge Files
Project-specific; populated by `/onboard` and maintained during development work.

| | Element | Description |
|-|---------|-------------|
| ⬜ | `.claude/knowledge/context/project-context.md` | Project description, primary users, business constraints, external dependencies |
| ⬜ | `.claude/knowledge/context/architecture.md` | System boundaries, component map, key technical decisions, known debt |
| ⬜ | `.claude/knowledge/coding-guide.md` | Language-specific rules generated by `/onboard` (style, testing, linting, patterns) |

## Specification Files
Living project state; stored under `.claude/knowledge/health/`.

| | Element | Description |
|-|---------|-------------|
| ⬜ | `.claude/knowledge/context/requirements.md` | Full inventory of functional requirements with status (implemented / planned / deferred); source from which domain model, invariants, and ADRs are derived |
| ⬜ | `domain-model.md` | Canonical entities, concepts, and their relationships |
| ⬜ | `invariants.md` | Properties that must always hold; referenced by all roles |
| ⬜ | `fidelity-index.md` | Per-item coverage ratings (THOROUGH/MODERATE/LOW/NONE) from Sweep runs |
| ⬜ | `findings.md` | Log of adversary findings and auditor sign-offs |
| ⬜ | `escalations.md` | CRITICAL/HIGH findings or blocked work awaiting resolution |
| ⬜ | `.claude/knowledge/deferred.md` | Backlog of intentionally deferred work; read by Claude to inform future sessions |
| ⬜ | `assumptions.md` | Recorded assumptions that bear on design decisions |
| ⬜ | `ubiquitous-language.md` | Agreed terminology used consistently across all outputs |
| ⬜ | `failure-modes.md` | Documented ways the system can fail (FM-NNN entries) |
| ⬜ | `.claude/knowledge/context/adr/README.md` | Index of all architecture decision records |
| ⬜ | `.claude/knowledge/context/adr/0001-template.md` | Template for new ADR files; must include an Acceptance Criteria section so auditor can verify intent, not just tests |

## Guardrails
Always-on constraints; no mode or role overrides them.

| | Element | Description |
|-|---------|-------------|
| ⬜ | Security guardrail | Handling of credentials, PII, and sensitive files; security findings reporting |
| ⬜ | Operations guardrail | Confirmation required before destructive git, filesystem, database, or external operations |

Structural frameworks referenced by modes and roles.

| | Element | Description |
|-|---------|-------------|

| ⬜ | Checkpoint protocol | Two-tier verified commit procedure: Tier 1 (spec changes only), Tier 2 (implementation changes) |
